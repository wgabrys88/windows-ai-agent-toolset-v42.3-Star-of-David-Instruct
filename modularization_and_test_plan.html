1 · Executive Summary
Feasibility
High. The codebase is small (~1 200 lines across 4 Python files). Full modularization is achievable in 2–3 focused sessions with zero external dependencies beyond stdlib + ctypes.
Biggest Wins
  • Extract coordinate math → pure, testable module
  • Replace global _STATE god-object with typed dataclass
  • Decouple VLM calls from HTTP handler
  • Make brain a pure function (no global _completed_labels)
Biggest Risks
  • win32.py calls ctypes at import time — breaks on non-Windows
  • Race conditions in _STATE lock + SSE subscriber list
  • Brain global state causes incorrect benchmark replay
  • CONDUCTOR output parsing is fragile (turns.txt proves it)
Root Cause of the turns.txt Failures

Two distinct bugs are visible in the benchmark log:

CRITICAL Invalid drag(1000,629…)

Turn 1: VISIONARY described the cursor label overlay text "[1000,629]" as if it were canvas content. CONDUCTOR hallucinated a drag using those pixel-label numbers verbatim. _parse_drags accepted drag(1000,629, 1000,629) because 1000 ≤ 1000 passes the bounds check. The drag was a zero-length no-op at a wrong position.

turns.txt Turn 1 CONDUCTOR output: drag(1000,629, 1000,629)
CRITICAL Premature DONE state

Turn 2: CONDUCTOR output contained 8 drag lines followed immediately by the word DONE on the same response. _is_done only checks the full stripped string, so it returned False and the drags were executed. But the verify-call (≥3 drags path) then received a fresh image and returned DONE, which forced _completed_labels = all strokes regardless of what was actually drawn. The loop then short-circuits on every subsequent turn.

turns.txt Turn 2 CONDUCTOR output (truncated): drag(500,900, 850,300) ... drag(850,700, 500,100) DONE ← mixed in same response; verify call also returns DONE

3 · Detailed Audit Findings
3A · Dead / Unreachable Code
DEAD CODE franz.py:68–100 — Action builder helpers never used by brain_bench

click(), double_click(), right_click(), type_text(), press_key(), hotkey(), scroll_up(), scroll_down(), drag_start(), drag_end() — all return dicts that must be put into action_queue manually. brain_bench.py never calls any of them; it calls subprocess win32 drag directly. These helpers exist for other brains but are untested and undocumented.

DEAD CODE franz.py:103–120 — dot(), box(), line() overlay helpers

Never called anywhere in the current codebase. Overlay dicts are constructed inline in brain_bench.py:_queue_target_overlays().

DEAD CODE router.py:_emit_swarm()

Thin wrapper around franz.swarm_message() with identical signature. Called in exactly one place (call_vlm). Zero abstraction value; adds indirection.

DEAD CODE brain_bench.py:CAPTURE_REGION / CAPTURE_WIDTH / CAPTURE_HEIGHT / CAPTURE_DELAY_SECONDS / ACTION_DELAY_SECONDS / SHOW_CURSOR / DEFAULT_CURSOR_POS

These config constants are defined in the brain but the engine loop in router.py reads them via franz.cfg(brain, ...). The brain itself never reads them directly — they exist purely as config carriers for the engine. This is an implicit contract with no interface definition.

DEAD CODE router.py:_STATE.brain_ann_live_b64

Set in do_POST /brain_annotated and exposed truncated in /state response, but never consumed by panel.html in any meaningful way (only first 20 chars sent). The full brain annotation flow (/brain_frame, /brain_annotated) is implemented but brain_bench.py never calls franz.request_annotation().

DEAD CODE brain_bench.py:CURSOR_COLOR / CURSOR_ARM / CURSOR_FONT_SIZE / CURSOR_LABEL_BG / CURSOR_SHOW_LABEL / CURSOR_LABEL_OFFSET / CURSOR_LABEL_LIMIT

Cursor appearance constants read by router.py:_make_cursor_overlay() via franz.cfg(). The brain never uses them itself. Same implicit-contract anti-pattern as above.

3B · Duplicated Logic
DUPLICATION VLM HTTP call logic duplicated in two places

router.py:call_vlm() and brain_bench.py:_call_agent() both construct identical JSON bodies, call urllib.request.urlopen, parse choices[0].message.content, and emit swarm messages. The only differences are: agent name ("VISIONARY" vs parameterized), and whether an image is always included. ~60 lines of near-identical code.

router.py call_vlm() vs brain_bench.py _call_agent(): Both: json.dumps({model, temperature, top_p, max_tokens, stream:False, messages:[sys,user]}) Both: urllib.request.Request(endpoint, data=body, headers=..., method="POST") Both: resp_obj.get("choices",[])[0].get("message",{}).get("content","") Both: franz.swarm_message(agent, "input"/"output"/"error", ...)
DUPLICATION Subprocess capture called from 3 different sites

router.py:_subprocess_capture(), brain_bench.py:_capture_fresh_b64(), and brain_bench.py:_latest_screenshot_b64() all deal with getting a screenshot. The first two spawn win32.py capture subprocesses with slightly different argument assembly. The third reads from disk. Three different strategies for the same logical operation.

DUPLICATION Coordinate normalization / region parsing duplicated

win32.py:_parse_region(), win32.py:_parse_pos(), win32.py:_norm_region_to_pixels(), win32.py:_norm_to_screen_pixel(), win32.py:_screen_pixel_to_norm() — all pure math functions buried inside the subprocess-only win32 module. router.py:_action_xy_str() re-implements the "get x,y from action dict" pattern. The region string format ("x1,y1,x2,y2" in 0–1000 norm space) is parsed in at least 4 places.

DUPLICATION Queue draining pattern repeated

franz.drain_queue() exists but router.py:_drain_swarm_into_state() calls it and then does additional processing. The same drain-then-process pattern appears in do_GET /state, do_GET /swarm, and engine_loop — 4 call sites with slightly different post-drain logic.

DUPLICATION Region string passed through every subprocess call

Every function in router.py that calls a subprocess (_subprocess_capture, _subprocess_cursor_pos, _subprocess_execute_one, _subprocess_execute_drag) independently calls franz.cfg(brain, "CAPTURE_REGION", "") and appends ["--region", region]. This 3-line pattern is copy-pasted 4 times.

3C · Tight Coupling & God Objects
GOD OBJECT router.py:_ServerState — 18 fields, no cohesion

Mixes: annotation pipeline state (raw_b64, pending_seq, annotated_b64, annotated_ready), brain annotation pipeline state (6 fields), display state (display_text, display_actions), swarm message buffer, phase/turn tracking, and error text. All protected by a single threading.Lock. Any change to any field requires acquiring the same lock, creating unnecessary contention.

GOD MODULE franz.py — entry point + bus + loader + config + helpers

Owns: 4 global queues, runtime overrides dict, all action/overlay builder functions, module loader, brain picker, region selector launcher, HTTP server startup, and the cfg() config accessor. Importing franz in a test pulls in all of this.

TIGHT COUPLING brain_bench.py calls franz.WIN32_PATH directly

The brain reaches into franz to get the path to win32.py and spawns subprocesses itself. This means the brain is coupled to: the filesystem layout, the subprocess protocol, and the franz module's internal path constants. It cannot be tested without a real filesystem and a Windows environment.

TIGHT COUPLING router.py accesses franz._brain_annotate_request and franz._brain_annotate_result directly

Private queues (underscore-prefixed) are accessed cross-module. The annotation bridge is split across franz.py (queue definitions + request_annotation()) and router.py (_check_brain_annotation_request(), POST handler). No single place owns the protocol.

3D · Race Conditions
RACE router.py:_wait_for_annotation() — seq check outside lock

The function sets _STATE.pending_seq inside the lock, then publishes frame_ready outside the lock. The panel could receive the event, fetch /frame, render, and POST /annotated before _publish_state() is called. The /annotated handler reads _STATE.pending_seq inside its own lock acquisition — this is safe, but the _EVENT_BUS.publish call at line ~310 reads _STATE.raw_seq without holding the lock.

RACE router.py:_EventBus.publish() — subscriber list mutation during iteration

Dead subscribers are collected into a dead list and removed inside the same lock that protects iteration. If put_nowait raises, the subscriber is marked dead and removed — but the removal calls list.remove() which is O(n) and could silently fail if the subscriber was already removed by a concurrent unsubscribe() call (caught by ValueError but still a TOCTOU).

RACE brain_bench.py:_completed_labels — global mutated from engine thread

_completed_labels is a module-level set mutated inside _run_swarm() which is called from the engine thread. If two turns somehow overlap (e.g. during the verify-call path where _execute_drags is called a second time), the set could be in an inconsistent state. No lock protects it.

3E · Bugs & Logic Flaws
BUG brain_bench.py:_parse_drags():L~195 — bounds check allows 1000 as valid

The check all(0 <= v <= 1000 for v in vals) accepts 1000 as a valid coordinate. The cursor label "[1000,629]" visible in the screenshot was parsed as drag(1000,629,1000,629) — a zero-length drag at the edge of the canvas. The label text leaked into the VLM context because the cursor overlay was rendered with coordinates outside the visible canvas area.

BUG brain_bench.py:_is_done():L~200 — does not handle mixed drag+DONE response

_is_done() strips <think> tags then checks if the entire remaining string equals "DONE". When CONDUCTOR outputs 8 drag lines followed by "DONE", _is_done() correctly returns False. But _parse_drags() silently ignores the "DONE" line and returns all 8 drags. The verify-call then sees a completed canvas and returns "DONE", which triggers _completed_labels = all strokes — even if the drags failed or were wrong.

BUG brain_bench.py:on_vlm_response():L~305 — blank-canvas detection is fragile

if not text or ("blank" in text.lower() and not _completed_labels) — returns the first-turn prompt only if the word "blank" appears in the VLM response. If the VLM says "empty", "white", "nothing", or "no strokes", the condition fails and the swarm runs with an empty visionary report.

BUG router.py:engine_loop():~L390 — previous_user_text is empty on turn 1

The first VLM call uses user_text_for_vlm = previous_user_text = "". An empty string is sent as the user message. call_vlm() substitutes a default "Observe the screenshot" text, but this substitution is invisible to the session log which records the empty string. The VISIONARY system prompt is also sent as the user-facing system prompt, which is correct, but the empty user text means the VLM has no task context on turn 1 — confirmed by turns.txt Turn 1 INPUT being blank.

BUG router.py:do_POST /annotated — seq mismatch returns 409 silently

If the panel sends an annotated frame with a stale seq (e.g. after reconnect), the engine blocks forever on annotated_ready.wait(timeout=300) and then falls back to the raw screenshot. The 409 response is logged nowhere server-side; the engine just times out after 5 minutes.

FLAW win32.py — ctypes.WinDLL(...).SetProcessDpiAwareness(2) at module import

Line ~90: DPI awareness is set as a side effect of importing the module. This makes win32.py impossible to import in any test environment (non-Windows, CI, etc.) and mutates process-level DPI state as a side effect of import.

FLAW brain_bench.py:_latest_screenshot_b64() — reads from disk instead of using live state

The CONDUCTOR agent receives the most recent PNG from the session log directory, not the current annotated frame held in _STATE.annotated_b64. This means CONDUCTOR sees a frame that may be 1–2 turns old if the filesystem write is slow, and it bypasses the annotation pipeline entirely.

def test_parse_drags_rejects_cursor_label_artifact():
    """Regression: turns.txt Turn 1 — cursor label '[1000,629]' leaked into VLM output."""
    raw = "drag(1000,629, 1000,629)"
    # After fix: 1000 is boundary — decide: reject or accept?
